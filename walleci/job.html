# 介绍

.walleci.yml中允许配置无限量的jobs。每个jobs必须有一个唯一的名字，job由一列参数来定义jobs的行为。

# 完整示例

jobs:
  - name: 构建  
    stage: test 
    image: harbor.hxsf.work/walle-ci/lighthouse:latest
    working_dir: default
    only:
      - master
    except:
      - develop
    allow_failure: true 
    before_script: 
      - rm -rf node_modules
    scripts:
      - npm install
    after_script:
      - ci-upload

# 配置参数说明

KeywordRequiredDescription
name必须给job命名
image非必须所使用的docker镜像，查阅使用docker镜像
script必须Runner执行的命令或脚本
stage非必须定义job stage（默认：test）
variables非必须定义job级别的变量
only非必须定义一列git分支，并为其创建job
except非必须定义一列git分支，不创建job
allow_failure非必须允许job失败。失败的job不影响commit状态
when非必须定义何时开始job。可以是on_success，on_failure，always或者manual
cache（待实现）非必须定义应在后续运行之间缓存的文件列表
before_script非必须重写一组在作业前执行的命令
after_script非必须重写一组在作业后执行的命令
entrypoint_args非必须类似 docker run [OPTIONS] IMAGE [COMMAND] [ARGS...]
working_dir非必须运行镜像时指定的工作目录
depends_on非必须指定job执行时依赖的额外job名称

# name(步骤名称)

给运行的job定义一个名称。

# stage(步骤归属)

stage允许一组jobs进入不同的stages。jobs在相同的stage时会parallel同时进行。查阅stages更多的用法请查看stages。

# image(运行镜像)

每个job运行在不同的容器中，image定义了使用的容器类别。 优先级：image(job) > image(pipeline)

警告

如此项在pipeline和job中都没有定义，则job会跳过步骤。

# script(执行脚本)

script是Runner执行的shell脚本。该参数可以用数组包含多个命令：

job:
  script:
    - uname -a
    - bundle exec rspec

有时候，script命令需要被单引号或者是双引号包裹起来。举个例子，当命令中包含冒号(:)时，script需要被包在双引号中，这样YAML解析器才可以正确解析为一个字符串而不是一个键值对(key:value)。使用这些特殊字符的时候一定要注意：:,{,},[,],,,&,*,#,?,|,-,<,>,=,!。

# before_script 和 after_script

它可能会覆盖全局定义的before_script和after_script：

before_script:
- global before script

job:
  before_script:
    - execute this instead of global before script
  script:
    - my command
  after_script:

# only and except

only和except是两个参数用分支策略来限制jobs构建：

  • only定义哪些分支和标签的git项目将会被job执行。
  • except定义哪些分支和标签的git项目将不会被job执行。

下面是refs策略的使用规则：

  • only和except可同时使用。如果only和except在一个job配置中同时存在，则以only为准，跳过except(从下面示例中得出)。
  • only和except可以使用正则表达式。

在下面这个例子中，job将只会运行以issue-开始的refs(分支)，然而except中设置将被跳过。

job:
  # use regexp
  only:
    - /^issue-.*$/
  # use special keyword
  except:
    - branches

# variables（Job级）

在job中是可以使用关键字variables来定义job变量。它的运行原理跟pipeline级别定义是一样的，但是它允许设置特殊的job变量。

当设置了job级别的关键字variables，它会覆盖全局YAML和预定义中的job变量。想要关闭全局变量可以在job中设置一个空数组：

job_name:
  variables: []
1
2

Job变量的优先级关系可查看pipeline的variables文档说明。

# allow_failure(允许错误执行)

allow_failure可以用于当你想设置一个job失败的之后并不影响后续的CI组件的时候。失败的jobs不会影响到commit状态。

当开启了允许job失败，所有的intents和purposes里的pipeline都是成功/绿色，但是也会有一个"CI build passed with warnings"信息显示在merge request或commit或job page。这被允许失败的作业使用，但是如果失败表示其他地方应采取其他（手动）步骤。

下面的这个例子中，job1和job2将会并列进行，如果job1失败了，它也不会影响进行中的下一个stage，因为这里有设置了allow_failure: true。

job1:
  stage: test
  script:
  - execute_script_that_will_fail # 执行失败的脚本
  allow_failure: true

job2:
  stage: test
  script:
  - execute_script_that_will_succeed # 执行成功的脚本

job3:
  stage: deploy
  script:
  - deploy_to_staging

# when(条件执行)

在执行stage中的job时，会判断上一个stage的状态是否能满足执行要求。

when可以设置以下值：

  • on_success - 只有前面stages的所有工作成功时才执行。 默认值。
  • on_failure - 当前面stages中任意一个jobs失败后执行。
  • always - 无论前面stages中jobs状态如何都执行。
  • manual - 手动执行(暂未开放)。

举个例子：

stages:
- build
- cleanup_build
- test
- deploy
- cleanup

build_job:
  stage: build
  script:
  - make build

cleanup_build_job:
  stage: cleanup_build
  script:
  - cleanup build when failed
  when: on_failure

test_job:
  stage: test
  script:
  - make test

deploy_job:
  stage: deploy
  script:
  - make deploy
  when: manual

cleanup_job:
  stage: cleanup
  script:
  - cleanup after jobs
  when: always

# entrypoint_args

使用该参数可以覆盖镜像中的entrypoint配置。注意此项配置高于script，配置后将不再执行script内容。

# working_dir

运行镜像时指定的工作目录，不填写默认是/walleci/code ，此项填写default，则不指定工作目录，

如果使用entrypoint_args，注意需要处理该项数据。

# depends_on

指定job执行时依赖的额外job名称，主要用于配置job运行时依赖基础环境。

警告

被依赖的job需要优先于当前job执行 当前job执行时，被依赖的job必须时运行状态，如果没有状态要求，可以用stage实现顺序执行

使用示例

version: 1

stages:
  - build
  - test
  - deploy

jobs:
  - name: Mysql服务
    stage: build
    image: harbor.hxsf.work/walle-ci/mysql:1.0.0
  - name: Redis服务
    stage: build
    image: harbor.hxsf.work/walle-ci/redis:1.0.0
  - name: build
    stage: build
    image: harbor.hxsf.work/walle-ci/maven:1.8
    scripts:
      - mvn -v
    depends_on:
      - Mysql服务
      - Redis服务

  - name: mongo依赖
    stage: test
    image: harbor.hxsf.work/walle-ci/mongo:1.0.0
  - name: test
    stage: test
    image: harbor.hxsf.work/walle-ci/maven:1.8
    scripts:
      - mvn -v
    depends_on:
      - Mysql服务
      - mongo依赖

  - name: deploy发布
    stage: deploy
    image: harbor.hxsf.work/walle-ci/deploy:1.0.0
